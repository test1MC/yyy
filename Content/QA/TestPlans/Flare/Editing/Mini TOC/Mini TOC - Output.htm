Support in Print Outputs:

Support in PDF Outputs:

Like the main TOC, the mini TOC in print outputs is based entirely on heading levels. So, regardless of the structure of your topics in the TOC Editor, if you do not have your headings set up appropriately, the mini TOC may not output as desired.

 

Test Cases: 

  • Basic Use Case: Organize your topics in the TOC Editor in the appropriate order (regardless of tree structure) being sure to use heading levels appropriately. This should include creating single topics with descending heading levels as well as creating descending heading levels from one topic to another
  • Target Editor "patch my headings" Use Case: Place topics in the TOC Editor with no regard for the heading level in the actual topics. In the Target Editor's "Printed Output" tab, enable the option to "Use TOC depth for heading levels." This option will redefine the first heading in each topic to correspond with its depth in the TOC Editor and relatively update any other headings included in those topics.

 

There is currently different behavior for native PDF, XPS and XHTML outputs when compared to Word and Frame (See bug numbers 32269 and 32270). In Word and Frame, if you have a second mini TOC within the range of headings covered by a previous mini TOC, the second mini TOC "cuts off" to prior mini TOC. For example, I have a mini TOC that occurs under an h1 and is set to display headings 2-9. I have another mini TOC that occurs after an h4 that falls inside the range of the first mini TOC. This second mini TOC will cause the first mini TOC to stop at h4 instead of continuing to h9. Mini TOCs in PDF, XPS and XHTML outputs do not get cutoff in this way.

Support in DITA Output:

  • Mini-TOCs are not supported in DITA output. See compile error below.

Compiler errors:

  • When compiling DITA output:
    MadCap:miniTocProxy found in file: "Topic.htm". It will be removed from the output.
