TOC Proxy - Styles

The styles detailed below all contribute to the look of the generated Table of Contents in the output (in place of the TOC Proxy). Of the styles outlined here, only MadCap|tocProxy will allow you to create unique classes. All other styles are generated styles used by all TOC Proxies regardless of class.

The styles applied here should be tested in all outputs, however, keep in mind that these styles only affect the TOC Proxy placed inside a topic. These styles do not affect the look of the TOC pane that is generated for Web, DotNet and HTML Help outputs. Styles for the TOC pane in Web, DotNet, and HTML Help outputs are set in the Skin and will be covered in the Skins test plan.

Other relevant styles: mc-heading-level

For all print output types, the generated TOC is built based on the headings you use throughout your topics. h1 elements will appear in the generated TOC as top-level entries. h2 elements will appear as second level entries and so on. This only works because h1, h2, h3 etc are assigned an "mc-heading-level" in the stylesheet.

 

h1, h2, h3, etc.

Assorted Relevant Properties

Default properties:

mc-heading-level: 1,2, etc (h1 = 1, h2=2, etc)

 

You can change the mc-heading-level of the h2 class to "3" and it will appear styled like a Heading 2 but it will behave like a Heading 3 when it comes to the generated print TOC. You can also set the mc-heading-level on any class, such as a class of p. This would make any p tag of that class appear in the generated TOC.

 

The most useful application of mc-heading-level is to set it to "0". By setting mc-heading-level to "0" Flare will not include that heading in the generated TOC. This is perhaps most commonly used for the heading used in the topic containing the TOC Proxy to prevent "Table of Contents" from appearing as an item in the generated Table of Contents
